canon-signal

An opinionated, trace-first observability toolkit for TypeScript. This tutorial assumes you've never thought about observability before. Read it once and you'll understand both the philosophy and the practice.

Start here

The 3am problem

It's 3am. You're the on-call engineer. PagerDuty just woke you up because the checkout endpoint is throwing 500 errors for 0.4% of requests. The dashboard shows the spike started 20 minutes ago. Your job is to figure out what's broken before the on-call shift ends.

You SSH into the production server and start tailing logs:

2026-04-09 03:14:22 Checkout started
2026-04-09 03:14:22 Cart total: 4999
2026-04-09 03:14:22 Checkout started
2026-04-09 03:14:22 User: usr_abc
2026-04-09 03:14:23 Payment provider response received
2026-04-09 03:14:23 Cart total: 12500
2026-04-09 03:14:23 Checkout failed: undefined
2026-04-09 03:14:23 User: usr_xyz
2026-04-09 03:14:23 Checkout completed

You can't tell which lines belong to which request. You can't tell which request failed or why. The error message says undefined. You don't know which user, which cart, which payment provider, which version of the code. You're guessing.

This is the problem observability solves. Specifically: observability is the discipline of producing your telemetry in such a way that 3am-you can actually answer questions, instead of guessing.

The mental shift: from logs to traces

The traditional approach is to log everything. Every interesting moment becomes a console.log call. The reasoning is "if I log enough, I'll be able to reconstruct what happened." This sounds reasonable until you try to do it at 3am.

The trace-first approach inverts this. Instead of scattered log lines, you produce one structured record per request. That record (called a root span) contains everything you might want to query later: who, what, when, where, how long, succeeded or failed. Sub-operations during the request become child spans, which give you a duration breakdown. The whole thing is called a trace.

Visually, a trace looks like this:

POST /checkout 200 847ms ├─ auth.verify 12ms ├─ db.query SELECT users 3ms ├─ payment.process 780ms │ └─ https.fetch stripe.com 750ms └─ db.insert orders 28ms └─ user.id=usr_abc customer.tier=enterprise

You read this top-down: a POST /checkout request returned 200 in 847ms. Inside it, auth verified in 12ms, a database lookup took 3ms, payment processing took 780ms (most of which was a Stripe API call), and the order was inserted in 28ms. The root span carries the user ID and customer tier as attributes.

If something went wrong, the trace would show the failed span in red, with the exception attached. You can immediately see where things broke and why, without guessing.

The mental shift is: traces are the source of truth, and logs are a backup. Anything you would have logged during a request should probably be an attribute on the trace's root span instead.

The people whose ideas shaped this

canon-signal isn't an original invention. It's the synthesis of ideas that have been refined for over 15 years across distributed systems research, production engineering, and the open-source telemetry community. Before we get to the code, here are the practitioners and papers whose thinking directly informs canon-signal's design.

Ben Sigelman, Yonatan Zunger, and the Dapper paper (Google, 2010)

Dapper, a Large-Scale Distributed Systems Tracing Infrastructure
Google Research, 2010 — the paper that started it all

Google's Dapper paper introduced distributed tracing as an industry concept. It described how Google instrumented every service across its production fleet so engineers could follow a single request from the load balancer through dozens of backends. The paper introduced the vocabulary we still use: trace, span, annotation, sampling. Every modern observability tool — Jaeger, Zipkin, Honeycomb, Datadog APM, OpenTelemetry — descends from Dapper.

Ben Sigelman went on to co-found LightStep (acquired by ServiceNow) and to be a co-author of the OpenTracing and OpenTelemetry standards. canon-signal builds on the Dapper model: traces are trees of spans, spans have parents and durations and attributes, and the whole tree describes one unit of work.

Brandur Leach (Stripe, 2016)

Brandur Leach — Canonical Log Lines
Stripe Engineering Blog, 2016

Stripe's engineering team observed that scattered log lines were impossible to query, so they invented "canonical log lines": one rich, structured log entry per request, containing every dimension you might want to filter or group by. User ID, customer tier, request path, response code, latency, payment provider, error code — all on one line per request.

The canonical log line was a workaround for the limitations of log-based observability before traces were widely available. The insight: if you can only have one observable artifact per request, make it as wide as possible.

canon-signal takes this idea seriously. Our root span is Stripe's canonical log line, with the structural superpowers of tracing layered on top. The attributes you set via signal.attr() are the canonical fields. The TypeScript interface that declares them is the schema Stripe wished they had.

Charity Majors and the wide-events crowd (Honeycomb, 2018+)

Charity Majors, Liz Fong-Jones, and the Honeycomb team
Honeycomb.io — wide events, "Observability 2.0"

Charity Majors has been the most prominent advocate for what she calls "Observability 2.0": the shift from monitoring known questions (CPU above 80%? error rate above 1%?) to asking arbitrary questions about your system's behavior. The vehicle for this is the wide event: a single record per unit of work containing dozens or hundreds of dimensions.

The argument against logs-first observability: logs are optimized for writing (easy to add a printf), not for querying. Wide events flip that — they're harder to write (you need to think about what dimensions matter) but trivial to query later. At 3am, querying matters more than writing.

canon-signal's typed schema is the "you need to think about what dimensions matter" step, enforced at compile time so you can't skip it.

Jeremy Morrell, Ivan Burmistrov, Boris Tane

Jeremy Morrell — A Practitioner's Guide to Wide Events
jeremymorrell.dev, 2024

Jeremy's guide is the most thorough practical treatment of how to actually implement wide events in a production codebase. He covers the schema design questions, the cardinality tradeoffs, the sampling implications, and the failure modes. canon-signal's middleware-and-attributes pattern follows his recommendations almost exactly.

Ivan Burmistrov — All You Need Is Wide Events
isburmistrov.substack.com, 2024

Ivan worked on Meta's internal observability system, Scuba, which is built entirely around wide events. His writing makes the case that wide events alone — without separate metrics or logs — can answer 95% of operational questions. canon-signal doesn't go quite that far (we still use metrics for infrastructure-style time series and logs for audit-level events) but the philosophy is the same: traces are the primary signal, and their attributes are the queryable surface.

Boris Tane — Logging Sucks
loggingsucks.com, 2024

Boris's site is a direct, concrete argument for why traditional logging fails at scale and how wide events fix it. It's the most accessible introduction to the philosophical shift if you're skeptical of "yet another observability framework." Worth reading if you want to be convinced before investing time in canon-signal.

The OpenTelemetry community (CNCF)

OpenTelemetry — the standardized SDK and protocol
CNCF, 2019–present

OpenTelemetry (OTel) is the open-source standard for emitting telemetry from applications. It's the successor to OpenTracing and OpenCensus, unified under one specification. Every major observability vendor (Honeycomb, Datadog, Grafana, New Relic, Splunk) supports OpenTelemetry, which means you can instrument your code once and send it anywhere.

canon-signal is built on top of OpenTelemetry, not as a replacement. We use the OTel SDK for span creation, context propagation, and export. What we add is opinion: a typed schema, automatic middleware, ambient context propagation, sampling rules, a test harness, and CLI tooling. If you outgrow canon-signal, the underlying spans are standard OTel and you can migrate to using the SDK directly.

Why canon-signal specifically

vs. raw OpenTelemetry

If you've ever tried to set up OpenTelemetry in a Node.js project, you know the experience. You start by installing thirteen packages:

npm install @opentelemetry/api @opentelemetry/sdk-node @opentelemetry/sdk-trace-base \
  @opentelemetry/sdk-trace-node @opentelemetry/exporter-trace-otlp-proto \
  @opentelemetry/resources @opentelemetry/semantic-conventions \
  @opentelemetry/auto-instrumentations-node @opentelemetry/sdk-metrics \
  @opentelemetry/exporter-metrics-otlp-proto @opentelemetry/sdk-logs \
  @opentelemetry/api-logs @opentelemetry/exporter-logs-otlp-proto

Then you write a hundred-plus lines of provider setup, processor configuration, and resource declaration. Then you finally get to call tracer.startActiveSpan() in your handler. And nothing in that setup tells you how to instrument well — what attributes to use, how to handle errors, how to keep span names low-cardinality, how to test that your instrumentation didn't drift in the last refactor.

Six months later, your codebase has 47 different ways of recording userId, half your handlers are missing critical attributes, your span names are high-cardinality time bombs, and nobody knows whether the instrumentation is even still working.

canon-signal: the opinionated layer

canon-signal's pitch is simple: one package, one factory call, an enforced schema, and automatic everything.

npm install canon-signal

That's the entire install. Then you write one setup file:

// src/signal.ts
import { createSignal, type SignalAttributes } from 'canon-signal'

interface AppAttributes extends SignalAttributes {
  'app.user.id'?: string
  'app.customer.tier'?: 'free' | 'pro' | 'enterprise'
}

export const signal = createSignal<AppAttributes>({
  service: { name: 'my-app', version: '1.0.0', environment: 'production' },
  schema: { version: '1.0.0' },
})

You now have a fully-configured observability system with:

  • A typed signal.attr() that won't compile if you misspell an attribute name
  • Ambient context propagation (no passing loggers around)
  • Auto-instrumentation for HTTP, databases, Redis, gRPC, and queues — all 37 OTel auto-instrumentations
  • A custom tail-sampling processor that always keeps errors and slow requests
  • A test harness with typed assertions
  • Pretty waterfall rendering in development, OTLP export in production
  • A processor that automatically computes app.db.total_duration_ms on every root span

The opinions are the value. canon-signal gets you to "correct telemetry by default" in one step instead of dozens.

Hands-on walkthrough

Let's build a real Hono app with canon-signal from scratch. Each step explains the why, not just the what.

Step 1: Install

mkdir my-app && cd my-app
npm init -y
npm install hono @hono/node-server canon-signal
npm install -D typescript @types/node tsx vitest

Create a minimal tsconfig.json:

{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true
  }
}

Step 2: Define your schema

Create src/signal.ts. This is the only place you configure canon-signal for your entire app.

// src/signal.ts
import { createSignal, type SignalAttributes } from 'canon-signal'

// Step 1: Declare the attributes your app will emit.
// The TypeScript interface is the contract. You cannot set
// an attribute that isn't listed here.
interface AppAttributes extends SignalAttributes {
  'app.user.id'?: string
  'app.customer.tier'?: 'free' | 'pro' | 'enterprise'
  'app.transaction.type'?: 'checkout' | 'refund'
  'app.error.code'?: string
}

// Step 2: Create the signal instance. This initializes the OTel SDK,
// registers auto-instrumentation, wires up the test harness, and
// returns a fully-typed object.
export const signal = createSignal<AppAttributes>({
  service: {
    name: 'my-app',
    version: '1.0.0',
    environment: process.env.NODE_ENV ?? 'development',
  },
  schema: { version: '1.0.0' },
  // Show a colored waterfall in the terminal during development
  export: {
    traces: [{ type: 'pretty-console' }],
  },
})

That's your entire observability setup. Note three things about it:

  1. The interface is the schema. Adding a new attribute later means editing this interface first. There's no runtime workaround.
  2. Optional vs required. The ? means optional. Non-optional properties are attributes that must be present on every root span (and the test harness enforces it).
  3. One signal per service. You export this from one file and import the same instance everywhere.

Step 3: Register middleware

Create src/server.ts:

// src/server.ts
import { Hono } from 'hono'
import { serve } from '@hono/node-server'
import { signal } from './signal'

const app = new Hono()

// Register canon-signal's middleware FIRST, before any routes.
// This is what creates the request scope — the magic that lets
// signal.attr() work from anywhere in your code without passing
// arguments around.
app.use('*', signal.middleware())

app.get('/hello', (c) => {
  return c.json({ message: 'Hello, world!' })
})

serve({ fetch: app.fetch, port: 3000 })
console.log('Server running on http://localhost:3000')

Run it:

npx tsx src/server.ts

And in another terminal:

curl http://localhost:3000/hello

In your server terminal you'll see something like:

GET /hello 200 3ms

That's your first trace, rendered as a waterfall. Right now it has no attributes — let's add some.

Step 4: Set attributes

Update the handler:

app.get('/hello', (c) => {
  signal.attr('app.user.id', 'usr_alice')
  signal.attr('app.customer.tier', 'enterprise')
  return c.json({ message: 'Hello, world!' })
})

Hit it again:

GET /hello 200 3ms └─ user.id=usr_alice customer.tier=enterprise

The attributes appear at the bottom of the trace. They're now on the root span and queryable in any backend.

Try this

Add signal.attr('app.bogus', 'value') to the handler. TypeScript will refuse to compile because 'app.bogus' isn't in your AppAttributes interface. This is the schema enforcing itself. The contract is the interface, and the compiler is the enforcer.

Step 5: Add a child span

Let's simulate a database lookup with a child span:

app.get('/hello', async (c) => {
  signal.attr('app.user.id', 'usr_alice')
  signal.attr('app.customer.tier', 'enterprise')

  await signal.span('db.user.lookup', async (span) => {
    span.setAttribute('db.user.id', 'usr_alice')
    await new Promise((resolve) => setTimeout(resolve, 50))
  })

  return c.json({ message: 'Hello, world!' })
})

Now the trace shows the child span's contribution to the total duration:

GET /hello 200 53ms └─ db.user.lookup 50ms └─ user.id=usr_alice customer.tier=enterprise

You can immediately see that 50ms of the 53ms total was the database lookup. If this request was slow, you know exactly where to look.

A few rules about child spans:

  • Use them for operations with meaningful duration. Don't wrap a 1ms function call in a span.
  • Keep names low-cardinality. db.user.lookup is fine. db.user.lookup.usr_alice is not — variable data goes in attributes, never in span names.
  • You don't need to wrap auto-instrumented operations. If you use pg or fetch, canon-signal already creates spans for them automatically. Wrapping them in signal.span() would produce duplicate spans.

Step 6: Handle errors

What if something fails? Let's deliberately throw an error:

app.get('/maybe-fail', async (c) => {
  signal.attr('app.user.id', 'usr_alice')

  try {
    await signal.span('risky.operation', async () => {
      throw new Error('something went wrong')
    })
  } catch (err) {
    signal.error(err)                                // records on the active span
    signal.attr('app.error.code', 'OPERATION_FAILED') // always targets root
    return c.json({ error: 'failed' }, 500)
  }

  return c.json({ ok: true })
})

Hit it and you'll see:

GET /maybe-fail 500 2ms └─ risky.operation 1ms └─ user.id=usr_alice error.code=OPERATION_FAILED

Red because it failed, with the error code surfaced as a queryable attribute. The exception details (type, message, stack trace) are recorded on the child span as a span event.

Note the deliberate split:

  • signal.error(err) targets the active span — in this case, the child span where the failure happened
  • signal.attr('app.error.code', ...) always targets the root span — the request-level outcome

Both annotations are preserved. When you debug, you see the error code on the root span (helping you filter "show me all OPERATION_FAILED errors in the last hour") and the actual exception on the child span (showing you exactly where it broke).

Step 7: Test the instrumentation

This is where canon-signal really pays off. Every signal instance has a built-in test harness with typed assertions.

Create src/server.test.ts:

import { describe, it, expect } from 'vitest'
import { Hono } from 'hono'
import { signal } from './signal'

const app = new Hono()
app.use('*', signal.middleware())

app.get('/hello', (c) => {
  signal.attr('app.user.id', 'usr_alice')
  signal.attr('app.customer.tier', 'enterprise')
  return c.json({ message: 'Hello, world!' })
})

describe('GET /hello', () => {
  const harness = signal.test.harness()

  it('emits a trace with the expected attributes', async () => {
    await app.request('/hello')

    const root = harness.rootSpan()
    expect(root).toBeDefined()

    // Typed assertions — the keys are constrained to your interface
    harness.assertAttr(root!, 'app.user.id', 'usr_alice')
    harness.assertAttr(root!, 'app.customer.tier', 'enterprise')

    // Structural assertion
    harness.assertStatus(root!, 'OK')

    harness.reset()
  })
})

Run it:

npx vitest run

Your test passes. You've just verified that the handler produces the right trace. If a future refactor accidentally removes the signal.attr('app.user.id', ...) call, this test will fail before the regression ships.

This is the superpower

Without the test harness, you'd only discover a missing attribute during a 3am incident — when an investigation comes up empty because the data isn't there. With it, you catch the bug at PR review time. Every handler with non-trivial instrumentation should have a test like this.

The constitution in practice

canon-signal is built on a set of principles documented in resources/CONSTITUTION.md (the file inside the package). Here are the most important principles, each tied to the code patterns you just saw.

Core principles

1. Traces are the backbone, not logs.

The canonical record of any request is its root span, not a collection of log lines. Logs are a secondary signal for the rare cases where you need a separate, correlated record outside the trace structure (audit trails, compliance events, system lifecycle).

In code: prefer signal.attr() over signal.log() for anything you'd want to query later. The default answer to "should I log this?" is "no, it should be an attribute."

2. The schema is the contract.

The TypeScript interface declares every attribute the service emits. The compiler enforces it. Drift is impossible — you can't accidentally misspell an attribute name or pass the wrong type, because the build will fail.

In code: interface AppAttributes extends SignalAttributes. This is the entire schema. Adding a new attribute means editing this interface first.

3. Context is ambient.

Every signal.attr() call automatically lands on the right request, no matter how deep in the call stack you are. This is enabled by Node.js's AsyncLocalStorage — the request scope propagates through all async operations without you passing arguments.

In code: notice that you never pass a "logger" or "request context" object to your service functions. They just call signal.attr() and it works.

4. Make the right thing easy, the wrong thing awkward.

canon-signal has two loggers: signal.log (which auto-attaches trace context) and signal.systemLog (which explicitly never attaches trace context). The split exists to force you to think about whether you're logging about a request or about the system. There's no single signal.log() that you can mindlessly call.

In code: if you find yourself reaching for signal.systemLog inside a request handler, that's a smell. The question to ask is "is this about the request?" — if yes, use signal.attr() instead.

5. Span names must be low-cardinality.

Trace backends index span names. High-cardinality names (with embedded variable data) blow up the index and break the backend. Names like payment.process are fine; names like payment.process.usr_${userId} are forbidden. Variable data goes in attributes.

In code: if your signal.span(name, ...) call has a template literal in the name, you're doing it wrong.

6. Sampling is outcome-aware.

canon-signal uses tail sampling: the decision to keep or drop a trace happens after the request finishes, when we know the outcome. Errors and slow requests are always kept; everything else is sampled at a configurable rate.

In code: configure sampling once in createSignal(). You never make sampling decisions in handler code.

Practical rules (the short version)

DoDon't
Use signal.attr() for queryable dataUse console.log in request handlers
Use signal.span() for sub-operations with meaningful durationWrap every function call in a span
Keep span names low-cardinality (templates only)Embed user IDs or query text in span names
Use signal.error(err) in catch blocksCatch errors silently

If you install canon-signal in your project, you can run npx canon-signal install-docs to copy the playbook (and the rest of the agent docs) into a .canon-signal/ directory at your repo root, where AI agents working on your code can read it.

Where to go from here

You now have enough to use canon-signal productively. The full feature set has more depth — sampling rules, metrics, logger bridges, the inspect CLI, agent debugging tools — but everything builds on the concepts you just learned.

Next steps in increasing ambition:

  1. Add a real database. Install pg or better-sqlite3. Make a query in a handler. Notice that auto-instrumentation creates a child span for your query automatically, with no code changes.
  2. Add sampling. Configure sampling: { alwaysKeep: { errors: true }, defaultRate: 0.5 } in your createSignal() call. Make 20 requests, fail one of them. Verify the failure is always kept and roughly half of the others are too.
  3. Run npx canon-signal install-docs in your project. Read the resulting .canon-signal/PLAYBOOK.md. Have any AI agents you work with read it too.
  4. Export to a real backend. Sign up for Grafana Cloud's free tier. Configure an OTLP exporter pointing at it. Watch your traces flow into Tempo in real time.
  5. Write tests for every handler. Use the harness to assert the attributes each handler should set. Run them in CI. Now telemetry quality is enforced like any other test.

The mental shift, restated

The whole point of canon-signal is to flip the question you ask when something goes wrong.

Old question: "What logs should I grep to figure out what happened?"

New question: "Let me open the trace and see exactly what happened."

If you take one thing away from this tutorial, take that. The traces are the source of truth. The schema is the contract. The tests verify the instrumentation. Everything else is implementation detail.
